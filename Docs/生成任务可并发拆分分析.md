# 生成任务可并发拆分分析

## 当前生成链路

当前主链路整体还是“单主任务串行推进”。

1. 前端在 [App.tsx](D:\NodejsP\pic-2-code\App.tsx) `handleGenerate` 中启动任务，拼接 `prompt` 和可选颜色信息，然后调用 `streamGenerateCode`。
2. 业务层在 [services/ai/service.ts](D:\NodejsP\pic-2-code\services\ai\service.ts) `buildGenerateRequest` 中把输入整理成单个模型请求。
3. 网关在 [server/ai/gateway.ts](D:\NodejsP\pic-2-code\server\ai\gateway.ts) 根据 provider 路由到 Gemini OpenRouter Claude 或 Codex。
4. 以 Codex 为例，在 [server/ai/codexRuntime.ts](D:\NodejsP\pic-2-code\server\ai\codexRuntime.ts) 中依次执行：
   - 写入临时图片文件
   - 构建完整提示词
   - 启动 `codex exec`
   - 等待模型完成
   - 读取最终输出文件

核心瓶颈在第 4 步。它像“一个主厨做整道主菜”，真正出菜前，大部分时间都被单个模型占住了。

## 可以直接并发的部分

### 1 预处理并发

这些任务彼此独立，适合在模型请求开始前同时跑：

- 图片颜色提取
  - 当前已经部分并发，见 [App.tsx](D:\NodejsP\pic-2-code\App.tsx) 中 `extractColors`。
- 图片基础分析
  - 如尺寸 纵横比 主体区域 透明背景检测。
- 提示词整理
  - 生成系统提示词 用户提示词 附加约束。
- Provider 健康检查或模型能力检查
  - 比如判断是否支持图片 是否需要本地网关。

这里适合用 `Promise.all`。收益不一定巨大，但实现成本低，能减少启动前的空等。

### 2 生成过程中的侧任务并发

主模型生成时，下面这些可以作为“副线程”并行跑：

- 日志流订阅与阶段展示
- 心跳检测与超时治理
- 隐藏预览容器预热
- 增量代码校验
  - 例如在收到流式代码片段后，后台检查 HTML 是否闭合 是否出现全屏遮罩 可疑动画脚本等。
- 轻量规则检查
  - 如是否只生成了要求的元素，是否混入多余区块。

这些任务不阻塞主模型，适合放到前端独立状态流，或浏览器 `Worker` 中。

### 3 生成完成后的派生任务并发

HTML 一旦稳定，很多后处理都可以一起跑：

- React 转换
- Flutter 转换
- React Native 转换
- 代码解释
- 渲染验收
  - 用隐藏 iframe 检查是否白屏 黑屏 报错
- 结果摘要
  - 例如提取页面结构 组件树 风格标签

这部分是最适合并发的。因为它们都依赖同一个最终 HTML，但彼此之间没有先后依赖。这里建议优先使用 `Promise.allSettled`，避免某一个失败拖死全部结果。

## 适合拆成双任务架构的部分

### 1 快速分析任务 加 主生成任务

这是最值得做的一层。

- 任务 A
  - 用快模型做截图理解，输出结构化结果
  - 例如布局分区 文本 OCR 颜色 组件清单
- 任务 B
  - 用强模型基于截图加结构化结果生成最终 HTML

严格说，B 最好等 A 的结果后再启动，所以它不是完全同时开始的并发，更像“前置快分析 加 主生成”。但如果 A 足够快，整体会比现在让大模型独自完成所有理解更稳。

这很像先让场记把镜头清单写好，再让导演正式开拍。导演仍然是主任务，但准备质量更高，返工更少。

### 2 主生成任务 加 后台验收任务

当主模型开始连续输出代码后，可以并发启动：

- DOM 结构检查
- CSS 风险扫描
- 运行时报错捕获
- 黑屏白屏检测

这样在主任务结束时，你就已经拿到第一轮验收结果，而不是等生成完再从头检查。

## 不建议并发的部分

- 同一个页面的核心 HTML 生成本体
  - 当前仍然依赖单个模型连续推理，不适合硬拆成多个并发子任务再拼接。
- 初次生成和精修同时进行
  - 精修依赖完整初稿，提前并发容易相互污染。
- 每次都多模型竞速
  - 虽然技术上可以做，但成本高，日志复杂，失败面更大。

## 建议的落地优先级

### 第一优先级 低风险高收益

- 生成完成后并发执行
  - 代码解释
  - 多端转换
  - 隐藏预览验收

### 第二优先级 中风险高收益

- 把“截图理解”拆成独立快任务
- 主模型只负责最终代码输出

### 第三优先级 高成本方案

- 多模型竞速
- 主模型和验收模型双通道联动

## 结论

如果只看你当前仓库，最适合并发拆分的不是“把 Codex 一次生成拆成几个并行 Codex 子任务”，而是把它外面那一圈辅助工作和后处理工作并发掉。

更具体地说：

- 现在就能做：预处理并发 后处理并发 验收并发
- 下一步值得做：快模型分析 加 强模型生成
- 暂时不要做：同一页面多模型乱并发拼接

## 类比理解

现在的流程像一个人独自完成“看需求 备料 炒菜 装盘 试吃”。

更合理的并发方式不是把“一盘菜”拆给三个人同时炒，而是：

- 一个人备料
- 一个人记单和盯火候
- 主厨专心炒主菜
- 出锅后另一人立刻试吃和摆盘

这样主厨不被杂事打断，整体吞吐反而更高。
